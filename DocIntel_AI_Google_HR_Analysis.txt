SECTION 2: CRITICAL BUGS & BROKEN DESIGN DECISIONS
--------------------------------------------------------------------------------

BUG #1 — REDIS FALLBACK IS A CONCURRENCY TIME BOMB
  Location: HITL Review Portal concurrency locking
  Issue: "Redis-backed session locker WITH in-memory dict fallbacks"
  Why it's a bug: In a distributed setup (even 2 worker instances), the
  in-memory dict fallback means Worker A and Worker B have SEPARATE dicts.
  Two reviewers CAN lock the same document simultaneously. Data corruption
  guaranteed under concurrent load.
  Fix: Redis must be the ONLY lock. Remove the fallback dict entirely.
  Use SET NX EX (atomic set-if-not-exists with TTL). Add lock heartbeat
  renewal so a reviewer doesn't lose their lock mid-edit.

BUG #2 — RABBITMQ CONSUMER HAS NO DEAD-LETTER QUEUE
  Location: worker.py (RabbitMQ consumer)
  Issue: If OCR crashes, Gemini times out, or the DB write fails mid-pipeline,
  the message is either lost (auto-ack) or requeued infinitely (nack loop).
  Fix: Configure a Dead-Letter Exchange (DLX). Failed messages after N retries
  go to a dead-letter queue for inspection, not into an infinite retry loop
  that hammers your OCR and Gemini services.

BUG #3 — CONSENSUS WEIGHTS ARE HARDCODED MAGIC NUMBERS
  Location: Multi-Agent Consensus Engine
  Issue: Critic=50%, Auditor=30%, Compliance=20% are baked in as constants.
  This means a compliance-critical legal document is scored the same as an
  invoice. A contract with bad math passes if Critic is happy. The weights
  must be DOCUMENT-TYPE AWARE.
  Fix: Build a WeightConfig schema per document category stored in the DB.
  Let the Classifier's output feed into weight selection before consensus runs.

BUG #4 — AUDITOR AGENT ZERO-SCORE OVERRIDE IS TOO BLUNT
  Location: Auditor Agent
  Issue: "If math fails, override score to 0.0" — this sends ALL math-broken
  docs to human review regardless of severity. A $0.01 rounding error in a
  $10M invoice gets the same treatment as a completely fabricated total.
  Fix: Grade math failures by magnitude. Add a tolerance_threshold config
  (e.g., delta < 0.5% of total = WARNING not FAIL). Assign a graduated
  score reduction instead of a binary zero.

BUG #5 — NO IDEMPOTENCY ON DOCUMENT UPLOADS
  Location: Ingestion route
  Issue: If a user uploads the same file twice (network retry, double-click),
  you'll process it twice, create two DB records, two ChromaDB embeddings,
  two audit log entries. Classic distributed systems mistake.
  Fix: Hash the file content (SHA-256) on ingestion. Check the hash against
  existing records before creating a new document entry. Return the existing
  document ID if it's a duplicate.

BUG #6 — TESSERACT OCR IS BLOCKING THE WORKER THREAD
  Location: services/ocr.py (assumed pytesseract usage)
  Issue: pytesseract.image_to_string() is a synchronous subprocess call.
  Running this inside an async FastAPI route or a RabbitMQ callback blocks
  the event loop or the consumer thread entirely.
  Fix: Wrap Tesseract calls in asyncio.run_in_executor() with a ThreadPoolExecutor
  in async contexts. In the worker, use a process pool for CPU-bound OCR work.

BUG #7 — JWT AUTHENTICATION HAS NO REFRESH TOKEN MECHANISM
  Location: PyJWT auth routes
  Issue: README mentions PyJWT for auth. Standard mistake: access tokens
  expire, no refresh token = users get logged out mid-review session, losing
  unsaved human-review edits.
  Fix: Implement a dual-token system. Short-lived access token (15 min) +
  long-lived refresh token (7 days) stored as HttpOnly cookie. Refresh endpoint
  rotates the refresh token on use (prevents replay attacks).

BUG #8 — CHROMADB IS RUNNING EPHEMERAL / IN-MEMORY BY DEFAULT
  Location: services/vector_store.py
  Issue: ChromaDB's default mode is in-memory. Every backend restart nukes
  ALL your vector embeddings. All indexed documents are gone.
  Fix: Configure ChromaDB with a persistent directory path:
  `chromadb.PersistentClient(path="./chroma_store")` and add this path
  to Docker volumes. OR migrate to a managed vector DB (Pinecone, Weaviate Cloud,
  Qdrant Cloud) for a production-credible story.

BUG #9 — THE POWERSHELL LAUNCH SCRIPT IS WINDOWS-ONLY
  Location: start_platform.ps1
  Issue: This immediately signals "I only tested on my own machine." Any
  Google engineer on macOS or Linux (which is everyone) cannot run this.
  Fix: Add a Makefile with `make dev` target. Add a shell script equivalent.
  Better: replace both with a proper docker-compose.override.yml that starts
  ALL services (backend + worker + frontend) in containers.

BUG #10 — NO INPUT SANITIZATION ON FILE UPLOAD
  Location: Ingestion route
  Issue: What file types are actually validated? MIME type spoofing is trivial.
  A user renames a .exe to .pdf and uploads it. Does your OCR pipeline crash?
  Does it execute? Does it store a malicious file?
  Fix: Validate MIME type using python-magic (not just file extension).
  Enforce max file size limits. Scan uploaded files in an isolated subprocess.
  Store uploads OUTSIDE the web root so they're never directly served.

SECTION 4: WHAT TO ADD — HIGH-IMPACT TECHNICAL UPGRADES
--------------------------------------------------------------------------------

UPGRADE #1 — REPLACE TESSERACT WITH GOOGLE DOCUMENT AI (or Azure Form Recognizer)
  Tesseract is a 30-year-old local OCR engine with poor layout understanding.
  Google Document AI's Form Parser extracts structured key-value pairs natively
  from invoices, receipts, and contracts WITHOUT a separate "Extractor Agent."
  This would:
  a) Make the pipeline dramatically more accurate
  b) Let you drop the Extractor Agent and simplify the consensus system
  c) Show GCP-native thinking (very relevant if applying to Google)
  If you want to keep Tesseract for the offline/free demo path, add Document AI
  as a pluggable backend via a strategy pattern: OCRBackend = Tesseract | DocAI

UPGRADE #2 — ADD APACHE KAFKA INSTEAD OF (OR ALONGSIDE) RABBITMQ
  RabbitMQ is a message broker. Kafka is an event log. The difference matters:
  - RabbitMQ: good for task queues (you have this right)
  - Kafka: good for event sourcing, audit trails, replay
  Your Audit Trails Ledger would be a PERFECT Kafka use case. Every status
  transition (INGESTED → PROCESSING → AWAITING_REVIEW → PROCESSED) becomes
  a Kafka event. You get: full replay capability, stream analytics, and a
  genuinely impressive distributed systems story.
  Even just ADDING a Kafka topic for audit events (while keeping RabbitMQ for
  tasks) demonstrates dual-broker knowledge that almost no student project has.

UPGRADE #3 — CONTAINERIZE EVERYTHING WITH KUBERNETES MANIFESTS
  Docker Compose is dev tooling. Google runs Kubernetes. Add:
  - Dockerfile for backend (multi-stage build: builder + slim runtime)
  - Dockerfile for frontend (build + nginx serve stage)
  - /k8s/ directory with Deployment, Service, ConfigMap, Secret YAMLs
  - HorizontalPodAutoscaler for the worker deployment
  - Liveness and Readiness probes on FastAPI (/health endpoint)
  You don't need to run it on real K8s — the manifests existing shows you
  know how production deployment works.

UPGRADE #4 — ADD STREAMING RESPONSES FOR LONG DOCUMENTS
  Currently the pipeline is fire-and-forget async (publish to RabbitMQ, poll
  for status). Add Server-Sent Events (SSE) or WebSocket for real-time status
  streaming:
  Document uploaded → SSE stream opens → 
    "OCR complete (2.3s)" → "Classification: INVOICE" →
    "Agent 1/4 complete" → "Consensus: 91.2%" → "Status: PROCESSED"
  This is the UX pattern Google products use. It transforms the dashboard from
  a polling page to a live feed. FastAPI has native SSE support via
  StreamingResponse.

UPGRADE #5 — ADD A DOCUMENT LINEAGE / PROVENANCE GRAPH
  Every transformation in your pipeline modifies the document's data. Right now
  you log events to an "Audit Trails Ledger" but there's no way to see:
  "The Critic Agent changed field 'vendor_name' from 'ACME Corp' to 'ACME Corp.'
  at 14:32:01 because it detected a trailing period mismatch."
  Add field-level diff tracking. Store before/after JSON for every agent pass.
  Expose this as a timeline view in the frontend. This is genuinely novel and
  directly mirrors how Google's internal ML pipelines track data provenance.

UPGRADE #6 — IMPLEMENT ACTIVE LEARNING FROM HUMAN CORRECTIONS
  Right now when a human reviewer corrects a field in the HITL portal, that
  correction vanishes into the DB and is never used again.
  Add: When a reviewer corrects extraction fields, log the (document_type,
  field_name, ocr_raw_text, wrong_value, correct_value) tuple. Build a
  corrections dataset. Fine-tune your TF-IDF extractor or use these examples
  as few-shot examples in your next Gemini call. This closes the feedback loop
  and gives the system a learning flywheel. This is ML engineering, not just
  AI integration.

UPGRADE #7 — ADD MULTI-TENANCY
  Enterprise document platforms are multi-tenant by design. Add:
  - Organization model in the DB
  - All documents, audit logs, and vector embeddings scoped to an org_id
  - Role-based access control: ADMIN, REVIEWER, VIEWER roles
  - API key management for programmatic document ingestion
  This transforms a demo into a credible SaaS architecture.

UPGRADE #8 — BENCHMARK AND PUBLISH ACCURACY METRICS
  The #1 thing that makes an AI project resume-worthy is NUMBERS.
  Run your pipeline on a labeled dataset (100 invoices, 50 contracts, 50 RFQs).
  Measure:
  - Classification accuracy (%)
  - Field extraction precision/recall per field per document type
  - Consensus score calibration (does 91% confidence actually mean 91% correct?)
  - Human review rate (what % of docs require human intervention)
  Put these numbers in your README. "91.3% field extraction accuracy on invoices,
  8.2% human review rate" is infinitely more impressive than "AI-powered platform."

9A. INTELLIGENCE LAYER — Make the AI Actually Smarter
────────────────────────────────────────────────────────────────────────────────

FEATURE: Cross-Document Contradiction Detection
  What: When a new document is uploaded, run a vector similarity search
  against ALL existing documents from the same vendor/counterparty. If the
  new invoice lists a unit price 40% higher than the last 5 invoices from
  the same vendor, flag it automatically as a PRICE_ANOMALY.
  Why it matters: This is a real enterprise use case (accounts payable fraud
  detection). No student project does this. It requires you to store
  per-vendor price history embeddings and run cosine similarity + statistical
  outlier detection (Z-score or IQR) on numeric fields.
  Tech: ChromaDB metadata filters + numpy stats + a new AnomalyDetector agent

FEATURE: Clause-Level Contract Intelligence
  What: For document type = CONTRACT/NDA/MSA, run a clause segmenter that
  splits the document into named clauses (Termination, Liability Cap,
  Governing Law, Indemnification). For each clause, run a separate
  classification + risk scoring pass. Output a "contract risk heatmap"
  showing which clauses are standard, modified, or missing vs. your
  internal template.
  Why it matters: This is literally what legal-tech startups charge $50k/year
  for. You'd be building a mini Harvey AI. The architecture is the same
  pipeline you already have — just with a new document type handler.
  Tech: spaCy for clause boundary detection + Gemini for risk classification

FEATURE: Named Entity Recognition (NER) Graph
  What: Across all processed documents, build a knowledge graph of entities:
  Companies, People, Dollar Amounts, Dates, Part Numbers, Legal Clauses.
  Store relationships in a graph structure (e.g., "ACME Corp" appears in
  47 documents, owes $2.3M, has 3 NDAs expiring in 2025).
  Expose a visual graph explorer in the frontend.
  Why it matters: This is the difference between "document storage" and
  "document intelligence." It's the feature that makes the platform feel
  like a living knowledge base, not a filing cabinet.
  Tech: spaCy NER + NetworkX for graph building + D3.js force-directed graph
  in the frontend

FEATURE: Temporal Intelligence — Expiry Alerts & Deadline Tracking
  What: Extract all dates from every document (contract end dates, payment
  due dates, warranty expiry, compliance certification dates). Store them
  in a unified timeline. Send proactive alerts (email/webhook) when a
  deadline is approaching (30 days, 7 days, 1 day).
  Why it matters: This turns DocIntel AI from a passive system into a
  proactive one. "Your NDA with ACME Corp expires in 14 days" is a feature
  CFOs actually pay for.
  Tech: dateparser library + APScheduler for cron jobs + SendGrid/Resend
  for email alerts (or just a webhooks endpoint)

FEATURE: Document Similarity & Deduplication Clustering
  What: Beyond upload-time SHA-256 deduplication, run semantic deduplication.
  After embedding, cluster documents by cosine similarity. If two invoices
  are 94% semantically similar but have different amounts, flag them as
  POTENTIAL_DUPLICATE with a diff view showing what changed.
  Why it matters: In enterprise settings, the same PO gets re-sent 3 times
  with minor edits. Catching this is high business value.
  Tech: ChromaDB vector search + DBSCAN clustering + unified diff display

SECTION 11: NEON DATABASE SETUP — CORRECT & SECURE CONFIGURATION

CRITICAL SECURITY NOTE:
  ⚠️  NEVER commit your Neon connection string to Git.
  ⚠️  NEVER paste it in chat, documentation, or any public place.
  ⚠️  ROTATE YOUR PASSWORD IMMEDIATELY at neon.tech → Settings → Reset Password.
  The connection string you shared is now compromised. A new one takes 30 seconds.

CORRECT SETUP:

Step 1: Store credentials ONLY in environment variables.
  Create backend/.env (add to .gitignore BEFORE creating):
  ---
  DATABASE_URL=YOUR_NEW_ROTATED_NEON_CONNECTION_STRING
  ---

Step 2: backend/app/config.py — correct Neon configuration
  ---
  from pydantic_settings import BaseSettings

  class Settings(BaseSettings):
      database_url: str
      # Neon requires SSL — ensure sslmode=require in the URL
      # channel_binding=require is Neon's security requirement
      # The -pooler. subdomain means you're using connection pooling (correct)

      class Config:
          env_file = ".env"

  settings = Settings()
  ---

Step 3: backend/app/database.py — SQLAlchemy engine for Neon
  ---
  from sqlalchemy import create_engine
  from sqlalchemy.orm import sessionmaker
  from app.config import settings

  # Neon is serverless — connections are expensive. Use pool_pre_ping to
  # detect stale connections. pool_size=5 is enough for dev.
  engine = create_engine(
      settings.database_url,
      pool_pre_ping=True,       # Test connection health before using
      pool_size=5,              # Neon free tier: max 10 connections
      max_overflow=2,           # Allow 2 extra connections under burst
      pool_timeout=30,          # Wait max 30s for a connection
      pool_recycle=1800,        # Recycle connections every 30 min
                                # (prevents Neon's serverless cold-start issues)
      connect_args={
          "sslmode": "require",
          "options": "-c channel_binding=require"
      }
  )

  SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
  ---

Step 4: Alembic migrations (stop using create_all)
  ---
  pip install alembic
  alembic init alembic
  # In alembic/env.py: set target_metadata = Base.metadata
  # Set sqlalchemy.url = your DATABASE_URL in alembic.ini
  # OR better: read from env var in env.py:
  #   config.set_main_option("sqlalchemy.url", settings.database_url)

  alembic revision --autogenerate -m "initial schema"
  alembic upgrade head
  ---

Step 5: CI/CD — migrate on deploy, never manually
  # In GitHub Actions deploy job, BEFORE starting the server:
  ---
  - name: Run database migrations
    run: |
      cd backend
      alembic upgrade head
    env:
      DATABASE_URL: ${{ secrets.DATABASE_URL }}
  ---

Step 6: Neon-specific production tips
  - Use the POOLER connection string (the one with -pooler. in the hostname)
    for your application. This routes through PgBouncer connection pooling.
    You already have this — the URL contains -pooler. which is correct.
  - Use the DIRECT connection string (without -pooler.) ONLY for Alembic
    migrations, because PgBouncer doesn't support all PostgreSQL features
    needed during migrations.
  - Enable Neon's "Auto-suspend" to save compute on free tier.
  - Set up Neon branching: create a "dev" branch for development and a
    "main" branch for production. Never run dev code against production DB.
    `neon branches create --name dev` — this is a VERY impressive thing to
    mention: "I use Neon database branching for environment isolation."
